## 1. Verdade de campo (o que realmente está conectado hoje)

Auditoria do código atual (não confiar no veredito por presença de símbolo):

| Função | Estado real | Onde |
|---|---|---|
| `buildInboxAccessFilter` | **USADA, mas incompleta** — chamada via `getInboxAccessFilter(userId, userPerfil)` **sem `effectiveSectors`** → cai no caminho "backwards-compatible" (legado). Inbox de suítes **não** é segregado por setor efetivo. | `useAccessFilters.ts:4`, `useInboxData.ts` (`useMemo`, síncrono) |
| `buildTaskAccessFilter` | **MORTA** (0 callers de produção). O Kanban segrega tarefas por uma cláusula **inline própria**, com semântica **diferente** desta função. | `SqliteKanbanRepository.getKanbanData:279-288` |
| `buildRecordAccessFilter` | **MORTA** (0 callers). O único read de records user-facing (inbox) usa a view `vw_inbox_normalizada` + `buildInboxAccessFilter`. `ListInboxUseCase`/`suites.query` não tem callers e não filtra acesso. | — |

### Consequências
- **Tarefas já são segregadas** (gap 3 resolveu inline) — 047.5 **não é** um buraco ativo de visibilidade de tarefas.
- **Buraco real remanescente:** inbox de suítes não respeita setor efetivo (gap 4 só foi marcado resolvido por presença de símbolo; o wiring de `effectiveSectors` **não** foi feito).
- **Risco de manutenção:** duas implementações divergentes da regra de acesso a tarefas (Kanban inline vs. `buildTaskAccessFilter`). Mudança em uma não propaga para a outra.

### Divergência semântica (Kanban inline × `buildTaskAccessFilter`)
Consolidar o Kanban na função **muda comportamento** — por isso é arriscado:

| Aspecto | Kanban inline (atual) | `buildTaskAccessFilter` |
|---|---|---|
| Tarefa com `setor_id IS NULL` | **visível** (`t.setor_id IS NULL OR t.setor_id IN (...)`) | **oculta** (`t.setor_id IN (...)` é falso p/ NULL) |
| Caminho vertical | `criado_por OR atribuido_para OR EXISTS(interessados) OR EXISTS(projetos_interessados)` | `criado_por OR atribuido_para OR u_criador.perfil IN (subordinados)` |
| JOIN exigido | nenhum | `JOIN usuarios u_criador ON u_criador.id = t.criado_por` |
| `interessados` | considerado | **não** considerado |

→ Troca cega quebraria visibilidade (esconde tarefas sem setor, remove path de interessados, exige novo JOIN).

## 2. Estratégia (faseada, por risco crescente)

### Fase A — Completar a segregação do inbox (alto valor, baixo risco) ✅ recomendada primeiro
O único buraco de segurança ativo. Surface contida.

**Mudanças:**
1. `useAccessFilters.ts` — tornar `getInboxAccessFilter` assíncrono e injetar `effectiveSectors`:
   ```ts
   import { getEffectiveSectors } from '@/src/infrastructure/persistence/SectorQueryUtils';
   export async function getInboxAccessFilter(userId: string, userPerfil: string, db: DbLike) {
       const eff = await getEffectiveSectors(userId, db);
       return buildInboxAccessFilter(userId, userPerfil, eff.length > 0 ? eff : undefined);
   }
   ```
   (`DbLike = { query }`; passar `container.sqlite`.)
2. `useInboxData.ts` — `accessFilter` deixa de ser `useMemo` síncrono e passa a resolver via `useEffect` (estado `accessFilter`), pois agora é assíncrono. Reexecutar a query quando `accessFilter` chegar. Tratar estado de loading inicial (não disparar query com filtro vazio que vaze tudo — usar `1=0` enquanto não resolve).

**Cuidado crítico:** o caminho com `effectiveSectors` usa `buildSectorFilter(eff, undefined, 'v.id_proprietario')` → exige que a view `vw_inbox_normalizada` exponha `id_proprietario`. **Verificar a coluna na view antes** (`grep id_proprietario` na definição da view); se o nome diferir, ajustar o `ownerColumn`.

**Risco:** se `effectiveSectors` vier vazio para usuário legítimo (ex.: `setor_principal_id`/`pai_id` ausentes — ver ADR-047 gaps 1/2, hoje tratados como falso-positivo de schema), o filtro com sector pode **esconder** o inbox inteiro. Mitigação: manter fallback `eff.length > 0 ? eff : undefined` (sem setor efetivo → caminho legado, não trava).

### Fase B — Consolidar tarefas em `buildTaskAccessFilter` (médio valor, alto risco)
Só após Fase A e **com testes de equivalência**. Objetivo: fonte única da regra de acesso a tarefas.

**Decisão de design a tomar antes (bloqueante):** qual semântica é a correta?
- Manter visibilidade de tarefas com `setor_id NULL`? (Kanban hoje mostra; a função esconde.)
- Manter o path de `interessados`? (Kanban usa; a função não tem.)

→ Provavelmente `buildTaskAccessFilter` precisa ser **estendida** para cobrir `setor_id IS NULL` e `interessados` antes de substituir o inline — senão a consolidação regride funcionalidade. Isso é mudança na própria função (afeta qualquer futuro caller).

**Mudanças (se aprovado):**
1. Estender `buildTaskAccessFilter` para opção `includeNullSector` e cláusula de interessados (parametrizável).
2. `getKanbanData`: adicionar `JOIN usuarios u_criador ON u_criador.id = t.criado_por`, substituir o bloco vertical+sector inline por `buildTaskAccessFilter(userId, perfil, eff)`.
3. Garantir que `isAdminMgr` continua bypass (`1=1`).

### Fase C — `buildRecordAccessFilter` (baixo valor hoje)
Não há read de records user-facing desprotegido além do inbox (já coberto pela Fase A). Opções:
- **C1 (recomendada):** marcar `buildRecordAccessFilter` + `ListInboxUseCase`/`suites.query` como caminho não usado; se for manter, adicionar teste unitário cobrindo a geração de SQL e documentar que será conectado quando houver listagem de suítes fora do inbox.
- **C2:** se existir (ou for criada) uma tela de listagem de pacotes/suítes que leia `SqliteSuiteRepository.query` diretamente, injetar o filtro lá. Hoje não existe → não conectar a vazio.

## 3. Estratégia de testes (vitest — `vitest.config.ts` já existe; hoje há 0 testes de access filter)

### 3.1 Unit — `AccessFilterBuilder` (puro, sem DB) — base obrigatória
Arquivo novo: `src/infrastructure/persistence/__tests__/AccessFilterBuilder.test.ts`
- `buildTaskAccessFilter`:
  - admin → `{ clause: '1=1', params: [] }`.
  - não-admin sem `effectiveSectors` → só cláusula vertical, params `[userId, userId, ...subordinados]`.
  - não-admin com `effectiveSectors` → `(${sector}) AND (${vertical})`, ordem de params = `[...sectorParams, ...verticalParams]`.
  - `effectiveSectors: []` (lista vazia) → `buildSectorFilter` retorna `1=0` (documentar: trava tudo — por isso callers passam `undefined` quando vazio).
- `buildRecordAccessFilter`: idem (com/sem setores; via `EXISTS` em `s.id_proprietario`).
- `buildInboxAccessFilter`: admin → `1=1`; sem `userId`/`perfil` → `1=0`; gerente legado vs. caminho com `effectiveSectors`; contagem de placeholders == params.
- **Invariante geral:** `clause.split('?').length - 1 === params.length` para toda combinação (evita desalinhamento de bind).

### 3.2 Integração — read paths (com SQLite em memória)
Reusar o padrão dos testes existentes (ex.: `src/application/widgets/__tests__`) com um SQLite real/fake e seed mínimo:
- **Inbox (Fase A):** 2 usuários em setores diferentes (A, B) + suítes de cada setor → operador do setor A **não** vê suíte do setor B; admin vê ambas; usuário sem setor efetivo → fallback legado (vê o que era visível antes).
- **Kanban equivalência (Fase B):** dataset fixo; comparar o conjunto de IDs retornado pela query **inline atual** vs. a query **com `buildTaskAccessFilter`**. A Fase B só entra se os conjuntos forem idênticos (ou a diferença for explicitamente aprovada). Casos-chave: tarefa `setor_id NULL`; tarefa só acessível por `interessados`; criador subordinado.

### 3.3 Regressão manual (sem deps de teste, no app)
- Login como `operador` setor A: inbox e kanban mostram só itens do setor A (+ atribuídas/criadas/interessadas conforme regra).
- Login como `admin`: vê tudo.
- Usuário sem setor: não fica com telas vazias.

## 5. Riscos & rollback
- **Telas vazias** por `effectiveSectors` vazio em usuário legítimo → mitigado pelo fallback `undefined`; teste 3.2 cobre.
- **Desalinhamento de params** (ordem sector×vertical) → coberto pela invariante 3.1.
- **View sem `id_proprietario`** → checagem no passo 2 antes de codar.
- **Regressão de visibilidade no Kanban** (Fase B) → guardada por teste de equivalência; Fase B é opcional e separável.
- **Rollback:** Fases A/B/C são commits independentes e reversíveis; Fase A não depende de B/C.

## 6. Decisões — TRAVADAS (2026-06-01)
1. **Escopo:** A + B + C completo.
2. **NULL-sector:** tarefa sem `setor_id` **permanece visível ao nível** (mantém semântica inline atual do Kanban). → `buildTaskAccessFilter` deve ser **estendida** para incluir `setor_id IS NULL` no caminho horizontal e preservar o path de `interessados`, garantindo equivalência exata antes de substituir o inline.
3. **`buildRecordAccessFilter`:** manter como API (C1) + testes; conectar apenas quando houver listagem de suítes fora do inbox (não conectar a vazio).

**Salvaguarda inegociável:** a Fase B só é mergeável após `npm i && npx tsc --noEmit && npx vitest run` verdes — em especial o teste de equivalência 3.2 (mesmos IDs antes/depois). Sem isso, B fica em branch.
